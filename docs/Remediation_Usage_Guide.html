Remediation & Fix Extraction Guide

This document provides extremely detailed scenario-based usage documentation focusing on the Extract Fix / Check Text and Apply Remediation processes. By following this guide, you can ensure your STIG remediation workflows are 100% reliable and deterministic across complex Windows and Linux air-gapped environments.

1. Overview of the Remediation Lifecycle

The STIG Assessor project allows compliance and systems engineers to fully automate their STIG hardening deployments through a highly structured 3-stage lifecycle:

1. Extraction (Export)

Parse raw DISA XCCDF STIG benchmarks to intelligently identify strings, scripts, and registry keys needed to fix vulnerabilities.

2. Execution (Deploy)

Execute generated Shell, PowerShell, Ansible, or standalone HTML Playbook exports directly on isolated target images.

3. Verification (Import)

Ingest structured JSON or CSV scan logs from the target back into STIG Assessor to automatically annotate .ckl files.

2. How the Fix Extraction Engine Works

The core extraction capability does not rely on rudimentary string matching. Instead, it utilizes an advanced regex lexer combined with semantic bounds context analysis. It evaluates STIG fixtext and check-content nodes and matches them against more than 12 independent code patterns (such as standard POSIX prompts # / $, Markdown code blocks, PowerShell Cmdlets, systemctl/apt sequences, Registry paths, and GPO definitions).

CLI
python -m stig_assessor --extract path/to/U_Windows_Server_2022_STIG_V3R1_Manual-xccdf.xml --outdir .\Fixes\
Platform Intelligence Details

The platform analyzer evaluates the extracted strings natively. If it detects Set-Mdp, reg.exe, or Get-ItemProperty it explicitly marks the required runtime as Windows. This guarantees that OS-specific scripts are exported accurately, regardless of the overarching title of the STIG benchmark.

3. Ensuring Cross-Platform Reliability in Air-Gaps

In highly regulated and disconnected environments, remediation pipelines are notoriously fragile. The STIG Assessor project mitigates this by guaranteeing deterministic behavior on both Windows and Linux without system-specific dependencies:

  • Environment Agnostic Core: All parsing and processing logic is restricted purely to standard Python libraries, thereby eliminating native C-bindings (like lxml natively built) that commonly cause pip installation failures on target platforms.
  • Robust UTF-8 Encodings: Cross-platform file I/O operations explicitly declare encoding="utf-8". This completely circumvents corruption during PowerShell string execution caused by Windows environments interacting with Linux-originated benchmarks.
  • Dependency-Free HTML Payloads: HTML Interactive Playbooks function identically off-network by fully embedding all base64-encoded bash/powershell scripts, CSS tokens, and JavaScript engines internally.

4. Scenario 1: Windows Server Over WinRM

Goal: Bulk remediate an air-gapped Windows Server 2022 golden image, dynamically extracting fixes and executing them.

Step 1: Extract Fixes on Management Node
Extract only high and medium severity items to prevent unnecessary regressions in your pipeline.

Powershell
stig-assessor --extract "U_Windows_Server_2022_STIG_V3R1_Manual-xccdf.xml" `
    --outdir "C:\Staging\Windows_Fixes" `
    --extract-severity high medium

Step 2: Transfer and Execute via WinRM Wrapper
Generate your results directly via an automated deployment shell. The resulting script execution produces a standard JSON payload reflecting vulnerability states.

Powershell
# Inside your WinRM automated installation routine
Invoke-Command -ComputerName "SRV01" -ScriptBlock {
    $OutPath = "C:\Windows\Temp\remediation_results.json"
    & "C:\Staging\Windows_Fixes\remediation.ps1" | ConvertTo-Json | Out-File $OutPath
}

5. Scenario 2: Linux Disconnected (Bash/Ansible)

Goal: Standardize a Debian 13 Golden Image by bulk-importing manual bash remediation results to update the baseline CKL checklist.

Step 1: Extract Base Configurations

Bash
python3 -m stig_assessor --extract Debian_12_STIG.xml --outdir /tmp/debian_fixes/
cd /tmp/debian_fixes
# Make generated scripts executable and run
chmod +x *remediation.sh && ./remediation.sh

Step 2: Generating Import Payload
Whether via custom scripting or standard toolchains, format the output of the execution logs to match the STIG Assessor JSON manifest standard: remediation_results.json.

JSON
[
  {
    "vid": "V-230221",
    "ok": true,
    "msg": "Executed chmod 0600 successfully",
    "out": "Files processed: 12",
    "err": ""
  }
]

Step 3: Auto-Update STIG Checklist
Apply the results from the raw JSON scan directly to an existing base .ckl manifest. This step takes the raw output and annotates the STIG artifact securely.

Bash
python3 -m stig_assessor --apply-results /tmp/remediation_results.json \
    --checklist /tmp/target.ckl \
    --results-out /tmp/target_updated.ckl
Status Auto-Conversion Pipeline

If "ok": true is declared in the JSON payload, the engine natively mutates the CKL vulnerability's STATUS element from "Open" to "NotAFinding". The msg, out, and error variables are smartly consolidated and Prepended into the Finding Details field explicitly tagged with a [AUTOMATED REMEDIATION] watermark.

6. Deep-Dive: Bulk Application Logic

The --apply-results operation works deterministically and guarantees idempotency. This is crucial for avoiding duplicate file mutations in automated environments:

  • Data Deduplication: When consecutive execution runs target the identical vid, the pipeline intelligently determines the most recent timestamp to dictate standard priority.
  • Granular Injection Modes: The behavior is strictly bounded via --details-mode [prepend | append | overwrite] parameters. Utilizing Prepend retains historically recorded manual notes within the CKL.
  • Strict Status Isolation: If integrating into CI pipelines strictly for log aggregation, pass --no-auto-status. This ingests all script outputs into finding details while mandating human QA to flip states from Open to Closed.

7. Integrating External Shell / 3rd-Party Scripts

You absolutely are not locked into executing exclusively extracted STIG Assessor scripts! The CLI accepts generic outputs from Nessus scripts, proprietary Bash utilities, or configuration pipelines like Puppet.

Comma-Separated Verification (CSV):
If your custom validation utility outputs logs in simplistic CSV patterns, the parser adapts natively. Target standard columns containing vid, ok (e.g. true/false/1/0/Pass/Fail), and msg.

CLI
stig-assessor --apply-results CustomScanExport.csv --checklist SRV1.ckl --out SRV1_Auto.ckl

8. The Standalone HTML Interactive Playbook

For scenarios where automated configuration management utilities are restricted, STIG Assessor dynamically produces a fully-styled, robust, zero-dependency .html Application representing all necessary configurations structured structurally by Category metrics.

Offline execution Context

The DOM inherently isolates operations avoiding memory leaks. Relies purely on secure local storage allocations restricting network callbacks entirely.

Secure Text Serialization

Features embedded Click-to-Copy capabilities. Extracted PowerShell commands are encoded in Base-64 pre-rendering preventing arbitrary quote escaping by browser renderers.
