Automated STIG Remediation

A comprehensive guide to utilizing the STIG Assessor's script extraction and bulk-import features to automate security compliance on Linux and Windows endpoints.

The STIG Assessor provides a highly deterministic pipeline for extracting STIG remediation steps directly from DISA XCCDF benchmarks, automatically synthesizing cross-platform execution scripts, and ingesting execution results directly back into your checklists.

⚡

Warning: Always test extracted scripts in a staging or development environment before running in production. Some findings inherently require manual intervention.

1. Extracting Fixes

The first step in the remediation lifecycle is securely extracting executable logic from STIG baselines. The --extract flag will automatically parse an XCCDF dataset and synthesize matching operational logic into Bash, PowerShell, Ansible, JSON, and CSV formats.

Basic Extraction Syntax

python STIG_Script.py --extract U_RHEL_9_STIG_V1R1_Manual-xccdf.xml --outdir .\Fixes\

Filtered Extraction

You can optimize targeting by filtering severities, passing an existing checklist to extract fixes only for Open findings, or filtering by Vulnerability IDs (VIDs):

python STIG_Script.py --extract U_Windows_10_STIG.xml ^
    --extract-ckl WORKSTATION01.ckl ^
    --status-filter Open Not_Reviewed ^
    --extract-severity high medium ^
    --outdir .\Fixes\
💡

Tip: You can append --script-dry-run to synthesize scripts in simulation mode, which generates the output scripts structure but replaces execution with safe Print/Echo commands instead. You can also append --enable-rollbacks to generate automatic Registry backup logic primarily for Windows modules.

2. Executing Remediation Scripts

After extraction, you will find fully automated remediation scripts deposited in the specified --outdir. Execute the script natively on the target host or deploy via centralized configuration management strategies.

Linux Execution (Bash)

Ensure you run the bash script from an elevated or Root context to securely apply OS-level hardenings. The bash execution harness protects operations explicitly utilizing set -euo pipefail architectures to guarantee deterministic operation logic across all air-gapped nodes.

sudo bash .\Fixes\remediate.sh

Windows Execution (PowerShell)

Invoke PowerShell natively as an Administrator. Executions synthesize localized artifacts securely mapping verification evidence to STIG configurations natively.

.\Fixes\Remediate.ps1

Script Output Characteristics:

  • The script dynamically creates an evidence/ folder containing sub-logs for each Finding ID validation sequence.
  • At completion, a master JSON result file (e.g., stig_results_YYYYMMDD_HHMMSS.json) will be synthesized in your root working directory. This is the file you will inject!

3. Bulk Importing Results

Once remediation routines are completed, ingest the generated JSON directly back into the .ckl or .cklb checklist file to finalize the operational verification loop via the --apply-results flag.

python STIG_Script.py --apply-results stig_results_20261105_120400.json ^
             --checklist WORKSTATION01.ckl ^
             --results-out WORKSTATION01_Updated.ckl

The import mechanism processes the deterministic results. Finding states that evaluate to successful remediation will be securely tagged with operational updates.

Real-World Scenarios

Scenario 1 Resolving Open Vulnerabilities Automatically

An auditor gives you a checklist with 55 "Open" findings. You want to extract scripts strictly for the exact 55 failures, run them automatically, and update the checklist to submit for Approval.

  1. Extract exactly the 55 open fixes.
    python STIG_Script.py --extract U_Ubuntu_STIG.xml --extract-ckl Audit.ckl --status-filter Open --outdir .\FixDir
  2. SCP remediate.sh to the Ubuntu server and run it. The script mitigates the configuration and generates stig_results_xxxx.json. SCP the JSON back to your workstation.
  3. Inject the resultant compliance mapping back!
    python STIG_Script.py --apply-results stig_results_xxxx.json --checklist Audit.ckl --results-out Completed_Audit.ckl

Scenario 2 Creating Deployment Hardening Templates

You need to integrate a fully hardened Windows Server 2025 golden image inside of a native Hyper-V build orchestration process automatically prior to system deployment in air-gapped data centers.

  1. You extract the entire framework for a fresh image.
    python STIG_Script.py --extract U_Windows_Server_2022_STIG_V3R1.xml --outdir .\Hardening
  2. Integrate Remediate.ps1 directly into your declarative Packer, Ansible, or MDT provisioning structure prior to Sysprep encapsulation phases.

Troubleshooting

Issue Resolution
Script fails halfway through execution, skipping checks. Generated scripts correctly encapsulate failure blocks via the built-in conditional logic to circumvent rigid set -e failure events organically while maintaining logging precision. Verify specific commands manually inside the evidence/ trace.
No JSON is generated post PowerShell process. Double-check ExecutionPolicy in Windows limits. Confirm script execution generated STDOUT logging context sequentially and verify transcript locations precisely.
Remediation failed to change Open status in checklist. Check your execution JSON output trace. If the shell script exited with a failure on specific commands, it passes false systematically, meaning the JSON records standard compliance failures purposefully avoiding accidental auto-closures!
