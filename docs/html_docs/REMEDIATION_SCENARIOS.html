Extraction & Remediation

Complete Guidance for Automating the STIG Assessment Lifecycle
✓ Cross-Platform Compatibility Verified The remediation generation logic operates deterministically across both Windows and Linux, utilizing robust string extraction protocols to ensure outputs are reliable even in strict air-gapped environments.

This playbook establishes standard operating procedures for isolating CLI commands hidden within massive generic XCCDF benchmarks, translating them into executable `.sh` or `.ps1` code blocks, and securely rolling them out to compliance targets.

Scenario 1: Baseline Extraction & Script Synthesis

In a standard pipeline, auditing organizations provide raw benchmarks such as U_MS_Windows_11_STIG_V1R10.xml. This XML structure is vast, human-readable, but not machine-executable. We use the Assessor to transform it.

Step 1 Generate Remediation Scripts

Invoke the core extractor to parse the XML, locate Fix command blocks, and generate execution scripts for the target platform. Provide an output directory where artifacts will be deposited.

python -m stig_assessor --extract U_MS_Windows_11_V1R10.xml --outdir win11_fixes

Artifacts Produced:

FileDescription
Remediate.ps1Complete PowerShell script equipped with evidence generation and outcome parsing. Generated with strict idempotency loops.
remediate.shComplete Bash equivalent specifically generated when Linux targets are detected within the generic XML.
remediate.ymlAn Ansible playbook translating scripts into node orchestration tasks for datacenter deployment.
fixes.jsonRaw JSON mappings of extracted V-IDs to their parsed terminal commands for custom API ingestion.
Security Control: Dry-Run Mode Do not execute generated scripts directly in a production enclave without initial review. Use the --script-dry-run flag during extraction. This ensures the .sh and .ps1 output scripts print what they would execute without making system modifications.

Step 2 Filter by Security Severity (Optional)

If you only have authorization to remediate CAT I (High) vulnerabilities, append the severity filtering flag.

python -m stig_assessor --extract U_MS_Windows_11_V1R10.xml --outdir win11_high_fixes --severity high

Scenario 2: Surgical Extraction via CKL State

You rarely want to run a complete benchmark script against a production system if you have an active .ckl (Checklist) file representing its current state. You only want to run fixes for findings marked as Open.

Step 1 State-Aware Script Generation

Pass your existing CKL assessment into the extractor. It will cross-reference the XML, isolating commands exclusively for vulnerabilities documented as Open or Not_Reviewed.

python -m stig_assessor --extract U_MS_Windows_11.xml --extract-ckl SRV01_Scan.ckl --outdir srv01_remediation

The resulting Remediate.ps1 script will now bypass NotAFinding rules to prevent system instability.

Scenario 3: Target Execution & Evidence Generation

Once generated, transfer the script structure (Remediate.ps1 or remediate.sh) to your target machine.

Ensure PowerShell is executed with Administrator privileges. Execution policies must allow local script runs.

Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\Remediate.ps1

PowerShell Execution Features:

  • Pre-Fix Checks: Evaluates system state prior to mutation, trapping the check condition to evidence\V-ID_out.log.
  • Idempotent Command Capture: Securely invokes the parsed command. Outputs flow directly into the evidence directory rather than crashing pipelines.
  • Post-Fix Verifications: Re-evaluates check blocks to prove to auditors the execution successfully resolved the vulnerability.

Ensure Bash has sufficient privileges. Execution scripts are strictly written utilizing set -euo pipefail to prevent lateral failure consequences.

sudo chmod +x remediate.sh
sudo ./remediate.sh

Bash Execution Features:

  • Intelligent Redirection: Automatically maps STDOUT and STDERR directly to the evidence/ array for logging.
  • System Subshells: Blocks execute in isolated subshells to prevent environment variable contamination.

Scenario 4: Validating Success & Bulk Injection

After executing the remediation scripts, an artifact named stig_results_YYYYMMDD_HHMMSS.json is created mapping the exit codes and completion states. This allows a seamless pipeline back into the CKL.

Automatic Ingestion

Use the Assessor's Results Importer to mutate your `.ckl` file to automatically flip Open statuses to NotAFinding based on script telemetry.

python -m stig_assessor --apply-results stig_results_20260416...json --checklist SRV01_Scan.ckl --results-out SRV01_Final.ckl
  • Intelligent Overrides: Success injects standard auditor justifications into the Finding Details section.
  • Traceable Modifications: All injected artifacts prefix timestamps automatically.
